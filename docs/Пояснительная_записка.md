# QuizMaster — Пояснительная записка к проекту

**Интерактивная платформа для проведения квизов в реальном времени**

---

## 1. Общая характеристика проекта

QuizMaster — это веб-приложение (MVP) для создания и проведения квизов в режиме реального времени. Платформа позволяет организаторам создавать опросы с вопросами различных типов, запускать сессии с подключением участников по коду комнаты и отображать результаты в виде таблицы лидеров.

**Ключевые возможности:**
- Регистрация с разделением ролей (организатор / участник)
- Создание квизов с 4 типами вопросов (текст / изображение, одиночный / множественный выбор)
- Подключение к активному квизу по 6-значному коду комнаты
- Синхронная демонстрация вопросов с обратным отсчётом через WebSocket
- Автоматический подсчёт баллов и рейтинг в реальном времени
- Личный кабинет с историей участий и статистикой

## 2. Обоснование выбора средств разработки

### 2.1. Серверная часть — Node.js + Express

| Компонент | Выбор | Обоснование |
|-----------|-------|-------------|
| **Runtime** | Node.js | Единый язык (JavaScript) для клиента и сервера ускоряет разработку и устраняет контекстное переключение. Асинхронная событийная модель идеально подходит для real-time приложений |
| **Фреймворк** | Express.js | Самый распространённый и лёгкий веб-фреймворк для Node.js. Богатая экосистема middleware |
| **База данных** | SQLite (better-sqlite3) | Не требует установки и настройки сервера БД — идеально для MVP. Синхронный API упрощает код. Файловая база легко переносится |
| **Real-time** | Socket.IO | Де-факто стандарт для WebSocket на Node.js. Автоматическое переподключение, комнаты (rooms), fallback на polling при отсутствии WebSocket |
| **Аутентификация** | JWT (jsonwebtoken) | Stateless-токены — не требуется хранить сессии на сервере. Простая интеграция с WebSocket (передача токена при подключении) |
| **Валидация** | express-validator | Декларативная валидация входящих данных, встроенная в цепочку middleware |

### 2.2. Клиентская часть — React + Vite + Tailwind CSS

| Компонент | Выбор | Обоснование |
|-----------|-------|-------------|
| **Фреймворк** | React 18 | Компонентный подход ускоряет разработку сложных интерфейсов с множеством состояний. Огромная экосистема и сообщество |
| **Сборщик** | Vite | Мгновенный холодный старт и HMR (Hot Module Replacement). На порядок быстрее Webpack в режиме разработки |
| **Стилизация** | Tailwind CSS | Utility-first подход: стилизация прямо в JSX без переключения между файлами. Готовая дизайн-система из коробки. Маленький production-бандл (~5KB gzip) |
| **Роутинг** | React Router v6 | Стандартное решение для маршрутизации в React-приложениях. Поддержка вложенных маршрутов и защищённых роутов |
| **HTTP-клиент** | Fetch API | Нативное API браузера, не требует дополнительных зависимостей |
| **Сокет-клиент** | socket.io-client | Официальный клиент Socket.IO, полная совместимость с сервером |

### 2.3. Почему не другие технологии?

| Альтернатива | Причина отказа |
|-------------|----------------|
| Next.js | Избыточен для SPA без SSR. Добавляет сложность настройки (серверный рендеринг, маршрутизация на базе файлов) |
| MongoDB | Требует отдельный сервер или Atlas-аккаунт. Реляционная структура (квизы → вопросы → варианты) естественнее в SQL |
| PostgreSQL | Требует установки и настройки сервера. SQLite достаточно для MVP и прототипа |
| Redux | Избыточен при небольшом количестве глобального состояния (только auth и socket). React Context покрывает потребности |
| SASS/Less | Tailwind CSS предоставляет более быстрый цикл разработки без написания отдельных CSS-файлов |

## 3. Основные этапы разработки

### Этап 1: Проектирование (1 день)

**Что сделано:**
- Написана полная спецификация системы в формате spec-as-code (`spec.md`) — единый документ, описывающий модели данных, API-контракты, WebSocket-события, UI-состояния и дерево маршрутов
- Спроектирована схема базы данных из 6 таблиц (users, quizzes, questions, options, quiz_sessions, participants, answers) с индексами и внешними ключами
- Определён протокол WebSocket-событий (11 типов событий с чёткой структурой payload)

### Этап 2: Разработка серверной части (1 день)

**Что сделано:**
- Реализована система аутентификации с JWT-токенами и ролевой моделью (organizer/participant)
- Созданы CRUD API для квизов и вопросов (4 эндпоинта для квизов, 4 для вопросов)
- Реализована система сессий с генерацией 6-значного кода комнаты (без O/0, I/1 для избежания путаницы)
- Разработан WebSocket-обработчик с полным жизненным циклом квиза: запуск → показ вопросов → приём ответов → подсчёт баллов → завершение
- Настроена серверная валидация времени ответа (late answers rejected)
- Автоматический таймер вопросов с досрочным завершением

### Этап 3: Разработка клиентской части (2 дня)

**Что сделано:**
- Создано 10 страниц: главная, вход, регистрация, панель управления, создание/редактирование квиза, комната организатора, подключение, комната участника, профиль
- Реализован защищённый роутинг с разделением по ролям
- Внедрён контекст аутентификации (AuthContext) и WebSocket-контекст (SocketContext) для глобального доступа
- Реализован полный цикл организатора: создание квиза → добавление вопросов 4 типов → запуск комнаты → управление показом вопросов → просмотр результатов
- Реализован полный цикл участника: ввод кода → ожидание → ответ в реальном времени → просмотр результатов
- Анимированный интерфейс: обратный отсчёт с цветовой индикацией, прогресс-бары, анимация появления вопросов

### Этап 4: Интеграция и тестирование (1 день)

**Что сделано:**
- Настроен прокси-сервер Vite для бесшовной работы клиента и сервера на одном хосте
- Проведено сквозное тестирование API: регистрация → создание квиза → добавление вопросов → создание сессии
- Проверена сборка production-бандла (28KB CSS + 257KB JS, 79 модулей, 0 ошибок)
- Настроен CORS и WebSocket CORS для безопасного кросс-доменного взаимодействия
- Проект задокументирован и готов к немедленному запуску
